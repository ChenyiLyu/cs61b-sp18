Lec.17 - Introduction to Asymptotic Analysis

Runtime Characterization

We want to somehow characterize the runtimes of the functions below to check whether an sorted array includes duplicates.

dup1

boolean dup1(int[] A) {
    for (int i = 0; i < A.length; i += 1) {
        for (int j = i + 1; j < A.length; j += 1) {
            if (A[i] == A[j]) {
                return true;
            }
        }
    }
    return false;
}

dup2

boolean dup2(int[] A) {
    for (int i = 0; i < A.length - 1; i += 1) {
        if (A[i] == A[i + 1]) {
            return true;
        }
    }
    return false;
}
The characterization should:
  • Demonstrate superiority of one over another
  • Be simple and mathematically rigorous

Comparing Algorithms

From tables above, we can say dup2 is better:

  • Needs fewer operations to do the same work: 50,015,00150,015,00150,015,001 vs. 10,00010,00010,000
  • Scales better in the worst case: (N2+3N+2)(N^2 + 3N + 2)(N2+3N+2) vs. NNN

Asymptotic Behavior

In most cases, we care only about asymptotic behavior, i.e. what happens for very large input size. e.g.:

  • Simulation of billions of interacting particles
  • Social network with billions of users
  • Logging of billions of transactions
  • Encoding of billions of bytes of video date

Algorithms which scale well (e.g. look like lines) have better asymptotic runtime behavior than algorithms that scale relatively poorly (e.g. look like parabolas).

e.g.: Parabolas vs. Lines

4 Simplifications:

  2. Pick a representative operation (a.k.a. the cost model)

  4. Ignore multiplicative constants

2 ~ 4 should be OK because we only care about the order of growth of the runtime (a.k.a. scalability)

Lec.19 - Big O, Big Omega, Amortized Runtime

Big O

Whereas Big Theta can informally be thought of as something like equals, Big O can be thought of as less than or equal:

  • N3+3N4∈Θ(N4)N^3 + 3N^4 \in \Theta(N^4)N3+3N4∈Θ(N4)
  • N3+3N4∈O(N4)N^3 + 3N^4 \in O(N^4)N3+3N4∈O(N4)
  • N3+3N4∈O(N!)N^3 + 3N^4 \in O(N!)N3+3N4∈O(N!)

R(N)∈O(f(N))R(N) \in O(f(N))R(N)∈O(f(N)) means there exists positive constants k2k_2k2​ such that:
R(N)≤k2⋅f(N)R(N) \leq k_2 \cdot f(N)R(N)≤k2​⋅f(N)
for all values of NNN greater than some N0N_0N0​ (i.e. very large NNN)

Big Theta vs. Big O

Big Theta only expresses exact order of growth for runtime in terms of NNN

  • If runtime depends on more factors than just NNN, may need different Big Theta for every interesting condition
  • e.g.: The best case runtime of fff is Θ(1)\Theta(1)Θ(1), whereas the worst one is Θ(N2)\Theta(N^2)Θ(N2)

We can just use Big O instead and avoid qualifying our statement at all:

  • The runtime of fff is O(N2)O(N^2)O(N2)

Big O Abuse

In the real world, Big O is often used where Big Theta would be more informative, e.g. saying "Mergesort is O(Nlog⁡N)O(N \log N)O(NlogN)"

  • True statement, but not as strong as saying "Mergesort is Θ(Nlog⁡N)\Theta(N \log N)Θ(NlogN)"

Despite lower precision, Big O is used way more often in conversation:

  • Important: Big O does not mean worst case ! Often abused to mean this.

The Usefulness of Big O

Big O is still an useful idea:

  • Allows us simpler statement
    • e.g.: "Binary search is O(log⁡N)O(\log N)O(logN)" instead of "Binary search is Θ(log⁡N)\Theta(\log N)Θ(logN) in the worst case"
  • Sometimes don't know the exact runtime, so use OOO to give an upper bound
  • Easier to write proofs for Big O than Big Theta
    • e.g.: Finding runtime of Mergesort, we can round up the number of input items to the next power of 2

Big Omega

Big Omega can be thought of as greater than or equal:

R(N)∈Ω(f(N))R(N) \in \Omega(f(N))R(N)∈Ω(f(N)) means there exists positive constants k1k_1k1​ such that:
k1⋅f(N)≤R(N)k_1 \cdot f(N) \leq R(N)k1​⋅f(N)≤R(N)
for all values of NNN greater than some N0N_0N0​ (i.e. very large NNN)

Why Big Omega ?

Two common uses:

  • Very careful proofs of Big Theta runtime
    • If R(N)=O(f(N))R(N) = O(f(N))R(N)=O(f(N)) and R(N)=Ω(f(N))R(N) = \Omega(f(N))R(N)=Ω(f(N)), then R(N)=Θ(f(N)R(N) = \Theta(f(N)R(N)=Θ(f(N)
    • Sometimes it's easier to show OOO and Ω\OmegaΩ separately
  • Providing lower bounds for the hardness of a problem
    • e.g.: The time to find whether an array has duplicates is Ω(N)\Omega(N)Ω(N) in the worst case for ANY algorithm (have to actually look at everything)

Potential

Key idea: Choosing aia_iai​ so that Φi\Phi_iΦi​ stays positive

For operation iii, choose an arbitrary amortized cost aia_iai​. This cost may be more or less than the actual cost cic_ici​ of that operation

    • Φi+1=Φi+(ai−ci)\Phi_{i + 1} = \Phi_i + (a_i - c_i)Φi+1​=Φi​+(ai​−ci​)
  • If we select aia_iai​ such that Φi\Phi_iΦi​ is never negative, then amortized cost is an upper bound on actual cost
